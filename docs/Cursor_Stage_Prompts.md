## Этап 2 — Расчёт метрик и статусов (светофор)

Цель: реализовать ядро расчётов, независимое от источников.

Модули: `src/metrics/occupancy.py`, `src/metrics/revenue.py`, `src/metrics/guests.py`.

Задача (формулы; unit = квартира, unit-night = квартира×ночь):
- Occupancy = продано unit-nights / доступно unit-nights × 100%.
- ADR = доход за проживание / продано unit-nights.
- RevPAR = доход / доступно unit-nights (или ADR × Occupancy).
- ALS = дней пребывания / кол-во броней.
- Доход: приоритет — факт (передаётся аргументом); fallback — snapshot-цена × занятые unit-nights, с флагом `is_estimated=True`.
- Светофор: функция принимает пороги из config и возвращает 🟢/🟡/🔴 для загрузки, отклонения цены, числа новых броней.
- Классификация каналов (`guests.py`): direct/aggregator по `channels_map` из config.
- Повторные гости: матчинг по телефону→email→ФИО (нормализация); идентификаторы хешировать.

Требования к коду: чистые функции без побочных эффектов; защита от деления на 0; никакой работы с сетью/БД.

Ожидаемый результат: `tests/test_metrics.py`, `test_channels.py`, `test_guests.py` покрывают формулы, граничные случаи (0 unit-nights, пустые данные), классификацию и матчинг гостей — все зелёные. PR «Этап 2».

## Этап 6 — Email: еженедельный HTML-отчёт + автосбор цен конкурентов

Цель: слать развёрнутый отчёт за неделю на email.

Модули: `src/notifiers/email_sender.py`, `src/data_sources/market_trends.py`, `src/data_sources/tl_ibe.py` + шаблон письма.

Задача:
- HTML-письмо (заголовки, таблицы) + текстовый дубль ключевых цифр. Отправка `smtplib` (SMTP из .env).
- Разделы: Occupancy по типам и в целом со сравнением к прошлой неделе; ADR и RevPAR (факт из TL или «оценочный» с пометкой); ALS; доля прямых vs агрегаторы; доля повторных гостей; тренды рынка (3–5 пунктов); конкуренты (цены на 7 дней, где доступно публично).
- Автосбор конкурентов (еженедельно, не в ежедневном цикле):
  - `static` → httpx + BeautifulSoup (Петровские, Гоголь, Кухтерин).
  - `tl_widget` → Playwright: вычислить `contextItemName` из HTML (regex), открыть виджет как пользователь, снять цены на 7 дней.
  - `wubook_widget` → Playwright: аналогично для WuBook (Xander).
  - Схема «DOM + скриншот» для виджетов:
    - цена берётся из DOM (основной канал);
    - скриншот сохраняется всегда (PNG) как доказательство;
    - vision-fallback: распознавание со скриншота, если цену из DOM получить не удалось.
  - Хранение скриншотов: `data/screenshots/competitors/YYYY-MM-DD/` (ретеншен 90 дней).
  - Основная функция: `parse_widget_with_screenshot()` в `src/data_sources/tl_ibe.py`.
  - Основной модуль — `src/data_sources/tl_ibe.py`, логика подстановки парсера — в `market_trends.py`.
- Dry-run работает и для email (слать себе/не слать).
- Список получателей — из config.

Требования к коду: вёрстка письма отдельной функцией; данные приходят из metrics/storage; запись в `reports_log`.
- Docker: в `docker/Dockerfile` установить Playwright + Chromium (для виджетов конкурентов).

Ожидаемый результат: письмо приходит на тестовый адрес корректно свёрстанным; функция генерации HTML покрыта тестом; при отсутствии факт-дохода стоит пометка «оценочный». Автосбор конкурентов отрабатывает по 3 стратегиям. PR «Этап 6».

## Этап 7 — TravelLine API (Universal + Read Reservation)

Цель: подключить эталонный источник фактических цифр и цен на даты.

Модули: `src/data_sources/travelline.py`.

Задача:
- Universal WebPMS API (хост `partner.tlintegration.com`): `analytics/services` (начисления/доход), `analytics/payments` (платежи), `analytics/services/cancelled`, `bookings/{number}`. Параметры: startDate/endDate (yyyyMMdd), dateKind. Для «новых броней сегодня» → `dateKind=2` (по созданию). Макс. окно 31 день.

- Авторизация: ключ из .env (заголовок Authorization/ключ из ЛК).
- Отдавай: факт-доход → фактические ADR/RevPAR; каналы → direct/aggregator (эталон); профили гостей → повторные; цены на конкретные даты.
- Сверка с Google Sheets: при расхождении выше порога (config) — предупреждение в сводку/лог.

Требования к коду: базовый URL/ключ — из config/.env; ретраи и обработка ошибок API; UTC↔MSK аккуратно.

Ожидаемый результат: авторизация работает; выгрузка новых броней по dateKind=2 и платежей возвращает данные; тесты с моками ответов API зелёные; расхождение с Sheets логируется. PR «Этап 7».

## Этап 10 — Обработка ошибок и уведомления о сбоях

Цель: сделать сервис предсказуемым при проблемах с источниками.

Модули: сквозные правки в data_sources/*, notifiers/*, scheduler.py + `errors_log`.

Задача:
- При временной недоступности источника — использовать последний успешный snapshot с пометкой «часть данных из последнего снимка» + техническое уведомление заказчику (Max/email).
- При недоступности критического источника (Google Sheets) — НЕ считать итоговые показатели полными; слать предупреждение вместо «красивого, но ложного» отчёта.
- Все ошибки/предупреждения — в `errors_log` и в раздел «Логи» админки.
- Единый механизм backoff/ретраев для сетевых вызовов.

Требования к коду: пометки в отчётах явные; уведомление о сбое отдельно от штатных сводок.

Ожидаемый результат: при симуляции падения источника (мок) сервис не падает, ставит пометки, шлёт уведомление, пишет в лог; тесты на деградацию зелёные. PR «Этап 10».
