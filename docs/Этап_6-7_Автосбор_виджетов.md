# Этапы 6/7 — автоматизация сбора цен с виджет-сайтов (без ручного ввода)

Документ для разработчика к ТЗ v2.2. Дополняет «Этап_6-7_Аудит_конкурентов.md». Задача: собирать цены со всех 9 конкурентов автоматически, включая 6 сайтов, где цены только в JS-виджете.

---

## Что выяснено технической разведкой

6 из 9 конкурентов используют один и тот же движок бронирования — **TravelLine IBE** (тот же, что у 1apart.ru). У каждого объекта есть постоянный идентификатор **contextItemName**, зашитый в код страницы:

| Конкурент | contextItemName (TravelLine) |
|-----------|------------------------------|
| Bon Apart | `TL-INT-bonapart.new` |
| Центральный | `TL-INT-centraltomsk` |
| Гоголь* | (статика, виджет не нужен) |
| Кухтерин** | `TL-INT-kuhterinhotel` |
| Скандинавия | `TL-INT-scandinaviahotel-ru_2023-08-04` |
| Парад Парк | `TL-INT-parad-park` |
| Элегант | `TL-INT-elegant-otel` |

\* Гоголь — цены в статике. ** Кухтерин — цены в статике, но виджет тоже есть.

Как извлечь contextItemName автоматически: скачать HTML главной конкурента (httpx) и найти regex-ом:
```python
import re
m = re.search(r"setContext['\"]?\s*,\s*['\"](TL-INT-[\w.\-]+)['\"]", html)
context = m.group(1) if m else None
```
Так `context` определяется динамически и не ломается при смене ID.

### Барьер
Хост виджета `ibe.tlintegration.com` / `ru-ibe.tlintegration.ru` при прямых серверных запросах отдаёт **nginx 403** (анти-бот защита: фильтр по заголовкам/поведению). Обычный httpx-запрос к API виджета не проходит. Значит, нужен подход, имитирующий реальный браузер.

---

## Рекомендуемая архитектура автосбора (гибрид, без ручного ввода)

Один общий сборщик для всех конкурентов на TravelLine + отдельные парсеры для статических сайтов.

### Стратегия по типу `parser`

1. **`static` (Петровские, Гоголь, Кухтерин)** — httpx + BeautifulSoup, как в основном ТЗ. Быстро и надёжно.

2. **`tl_widget` (Bon Apart, Центральный, Скандинавия, Парад Парк, Элегант)** — единый модуль `tl_ibe.py`:
   - Определить `contextItemName` из HTML сайта (regex выше).
   - Открыть страницу виджета в headless-браузере (**Playwright**, только для этих 5 объектов) с реальными заголовками, дождаться отрисовки списка номеров и снять цены из DOM виджета.
   - Playwright оправдан здесь именно потому, что API закрыто анти-ботом, а рендер виджета в браузере отдаёт цены штатно, как обычному посетителю.

### Почему Playwright, а не парсинг API
- Прямые запросы к API TravelLine IBE блокируются (403). Обходить защиту заголовками/прокси — хрупко и юридически спорно.
- Headless-браузер ведёт себя как настоящий гость: открывает виджет так же, как человек, и не «ломает» защиту.
- 1 snapshot в сутки × 5 сайтов = минимальная нагрузка, риск блокировки низкий при соблюдении анти-блок правил.

### Единый интерфейс сбора
```python
def collect_competitor_price(competitor, check_in, check_out) -> dict:
    if competitor.parser == "static":
        return parse_static(competitor)          # httpx + BeautifulSoup
    if competitor.parser == "tl_widget":
        ctx = detect_tl_context(competitor.url)  # regex по HTML сайта
        return parse_tl_widget(ctx, check_in, check_out)  # Playwright
    return {"available": False}
```

---

## Модуль `tl_ibe.py` (Playwright) — требования

- Один запуск браузера на весь прогон, последовательно по 5 объектам (не параллельно).
- Для каждого:
  1. `page.goto(competitor.url)` — открыть сайт конкурента.
  2. Кликнуть/инициировать виджет бронирования (кнопка «Забронировать» / `data-tl-booking-open`), либо сразу открыть URL виджета с параметрами дат.
  3. Дождаться селектора списка номеров (`networkidle` + ожидание элемента цены).
  4. Снять минимальную цену за ночь по датам (ближайшие 7 дней — как для своего объекта).
- Анти-блок: реалистичный User-Agent и viewport, задержки 3–5 сек между объектами, 1 прогон/сутки, `timeout` и повтор с backoff при неудаче.
- При неудаче (виджет не открылся/таймаут) → вернуть последний успешный snapshot из БД + пометка, НЕ падать.
- Результат: `{name, type, url, price_from, currency, check_in, check_out, source: "tl_widget", available: true}`.

### Даты запроса
Как и для собственного объекта — снимать цены на **ближайшие 7 дней** (фиксированный «утренний срез»), чтобы сравнение было сопоставимым.

---

## Зависимости и запуск Playwright

- Добавить в requirements (только для этой задачи): `playwright`.
- В Docker: установить браузеры `playwright install --with-deps chromium` в Dockerfile.
- Запускать сбор конкурентов **отдельной еженедельной задачей** (не в ежедневном цикле) — так Playwright не нагружает ежедневный snapshot и не влияет на скорость Max-сводки.

## Тесты
- static-сайты: фикстуры HTML, проверка извлечения цены.
- `detect_tl_context`: на сохранённом HTML каждого TL-сайта возвращает корректный `TL-INT-...`.
- `tl_widget`: интеграционный тест помечается `@pytest.mark.integration` (запускается вручную, требует Playwright и сети); в CI — только проверка graceful-фолбэка при недоступности.
