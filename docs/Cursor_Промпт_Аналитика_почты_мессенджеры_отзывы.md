# Cursor-промпт: расширение ИИ-аналитики (почты, мессенджеры, отзывы)

Промпты для разработки в Cursor по ТЗ `docs/TZ_analitika_pochty_messendzhery_otzyvy.docx`.
Задачи в GitHub: Issues #13–#17.

Как пользоваться: копируйте блок нужного этапа в чат Cursor (Agent). Выполняйте этапы по очереди. Cursor видит задачи на GitHub через MCP — можно просто просить «возьми Issue #13».

---

## Стартовый промпт (один раз в начале)

```
Ты — старший Python-разработчик проекта hotel-report-bot (отчётность и ИИ-аналитика для «Первого Апарт-отеля»).

Контекст проекта:
- Стек: Python, FastAPI, PostgreSQL/файловое хранилище, Playwright, YandexGPT.
- Уже реализовано: сбор данных (Google Sheets, сайт, TravelLine, тренды), расчёт загрузки/ADR/RevPAR, автосбор цен конкурентов, веб-кабинет с ИИ-лентой, сводки в Max и на email. LLM — YandexGPT в src/analytics/ai_insights.py.
- Проект работает в режиме dry_run=true (без реальной рассылки).

Задача — реализовать доработку по ТЗ docs/TZ_analitika_pochty_messendzhery_otzyvy.docx. Она разбита на 5 этапов (GitHub Issues #13–#17). Делаем строго по одному этапу за раз.

Обязательные правила:
1. ИИ только YandexGPT (существующий модуль), новых LLM-провайдеров не вводить.
2. Правовая модель: в текстах и выводах «квартиры/апартаменты», не «гостиничные услуги».
3. Секреты (токены, пароли) — только в .env и config, никогда в коде и в коммитах. Добавляй примеры в config/.env.example.
4. Личные данные гостей (телефоны, почты) не выводить в сводки — только агрегаты и обезличенные цитаты.
5. Отказоустойчивость: недоступный источник пропускается и логируется, сборка не падает. Для отзывов/переписки — кэш последнего успешного результата.
6. Всё должно работать в dry_run=true.
7. Каждый этап — отдельная ветка и Pull Request, с прохождением CI и деплоем на VPS.

Сначала изучи текущую структуру проекта (src/data_sources, src/analytics, src/web, config) и подтверди план по Этапу 1. Ничего не меняй, пока я не подтвержу.
```

## Этап 1 — Почты (Issue #13)

```
Возьми Issue #13 (Этап 1. Почты). Реализуй:
- src/data_sources/mail_inbox.py — подключение по IMAP, выборка писем за период (по дате), папки «Входящие» + метки.
- Классификатор писем: обращение / отзыв / отчёт сервиса / прочее (правила по отправителю и теме + YandexGPT для спорных).
- Парсер отчётов: адаптеры под TravelLine и типовые агрегаторы (по теме и телу письма), извлечь брони/отмены/суммы. Схему сделай расширяемой.
- Отзывы/претензии из писем помечай для передачи в блок отзывов (Этапы 3–4).
- Настройки в config/settings.yaml (ящики, папки, отправители-отчётов); секреты в .env: IMAP_YANDEX_*, IMAP_GMAIL_* или GMAIL_OAUTH_*. Обнови .env.example.

Проверка: письма читаются, отчёт TravelLine распознаётся, метрики попадают в общий поток. Работает в dry_run. Открой PR, дождись зелёного CI.
```

## Этап 2 — Мессенджеры (Issue #14)

```
Возьми Issue #14 (Этап 2. Мессенджеры). Реализуй:
- src/data_sources/messengers.py с адаптерами telegram.py, whatsapp.py, max_inbox.py — единый интерфейс fetch_messages(period).
- Telegram: Bot API (бот в чате поддержки) либо экспорт диалогов.
- WhatsApp / WhatsApp Business: Cloud API или провайдер (Wazzup / Green API).
- Max: чтение входящих (chat_id уже есть).
- Единая нормализованная структура сообщения: канал, дата, направление, обезличенный автор, текст, тема.
- Секреты в .env: TELEGRAM_BOT_TOKEN, WHATSAPP_API_*, MAX_TOKEN. Каждый канал — флагом в config. Обнови .env.example.
- Анализ: кол-во обращений по каналам, динамика, темы, тональность, доля повторных; отзывы/жалобы → в блок отзывов.

Проверка: обращения из подключённых каналов собираются и нормализованы, тональность считается, работает в dry_run. Открой PR.
```

```
Возьми Issue #15 (Этап 3. Сбор отзывов). Реализуй:
- src/data_sources/reviews.py + адаптеры reviews_2gis.py, reviews_yandex.py, reviews_google.py, reviews_aggregators.py.
- Источники: 2ГИС, Яндекс Карты/Справочник, Google Maps, агрегаторы (Островок, Яндекс Путешествия, 101hotels, Суточно, Твил).
- Сбор через Playwright + скриншот-фолбэк и распознавание при защите от автосбора — по аналогии с уже реализованным автосбором цен конкурентов.
- Дедупликация (автор + дата + хеш текста), инкрементальный сбор (только новые), хранение с полями source и object_id.
- Ссылки на карточки объекта и конкурентов — в config.competitors (объекты Томска уже есть).
- Вежливые задержки и лимиты; при блокировке — фолбэк на скриншот и повтор позже.

Проверка: отзывы по своему объекту и конкурентам собираются, дедуплицируются, работают инкрементально. Открой PR.
```

## Этап 4 — Классификация отзывов (Issue #16)

```
Возьми Issue #16 (Этап 4. Классификация). Реализуй:
- src/analytics/reviews_insights.py — классификация каждого отзыва через YandexGPT единым промптом.
- 3 категории: положительные / отрицательные / конструктивные.
- На выходе для каждого отзыва: категория + краткая суть (1 строка) + тег темы (чистота, заселение, цена, шум, персонал, локация, оснащение и т.п.).
- Кэш результата по хешу текста (не переклассифицировать повторно).
- Агрегаты: распределение по 3 категориям, средняя оценка, топ-темы жалоб и похвал — по своему объекту и по каждому конкуренту.

Проверка: каждый отзыв размечен, есть кэш, считаются агрегаты и сравнение с конкурентами. Открой PR.
```

## Этап 5 — Вывод результатов (Issue #17)

```
Возьми Issue #17 (Этап 5. Вывод). Реализуй:
Веб-кабинет — новый раздел «Отзывы и обращения»:
- Вкладка «Свой объект»: карточки по 3 категориям (счётчики + примеры), график динамики, топ-темы.
- Вкладка «Конкуренты»: таблица сравнения репутации, разбивка по категориям и темам.
- Вкладка «Каналы»: обращения из почты и мессенджеров (объём, темы, тональность).
- Блок «Конструктив → действия»: список предложений с кнопкой «в задачи/идеи».
Сводки:
- Ежедневная (Max/email): строка «новые отзывы по 3 категориям + число обращений по каналам».
- Недельная: блок «Репутация» — динамика, сравнение с конкурентами, 1–3 конструктивных вывода.
ИИ-лента: новые карточки «Репутация», «Отзывы конкурентов», «Конструктив недели».

Проверка: всё отображается, формулировки «квартиры/апартаменты», работает в dry_run. Открой PR. После мержа — финальный деплой.
```

## Порядок работы
1. Стартовый промпт → подтвердить план Этапа 1.
2. Этапы 1→5 по очереди: каждый = отдельная ветка + PR + зелёный CI + деплой.
3. После каждого PR — короткий отчёт: что сделано, какие доступы нужны от заказчика.
